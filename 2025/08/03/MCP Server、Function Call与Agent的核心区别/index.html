MCP Server、Function Call与Agent的核心区别

MCP Server

MCP Server 是大语言模型（LLM）获取外部能力和数据的一套标准化服务。它完全受控于调用方，只负责执行指令，不具备主动决策或推理能力。其功能涵盖多种工具，例如 Fetch MCP Server 用于网络抓取，而 Google Drive MCP Server 则可用于文件访问。简而言之，它是一个被动的执行者，而非主动的决策者。

sequenceDiagram
    participant User
    participant MCP-Client
    participant LLM
    participant MCP-Server
User->>+MCP-Client: 发送请求
MCP-Client->>+LLM: 传递用户查询及工具描述
LLM-->>-MCP-Client: 返回需调用的工具指令
MCP-Client->>+MCP-Server: 执行工具调用 (HTTP/SSE或stdio) 
MCP-Server-->>-MCP-Client: 返回执行结果
MCP-Client->>+LLM: 传递工具执行结果
LLM-->>-MCP-Client: 生成自然语言响应
MCP-Client-->>-User: 返回最终响应

例如我们基于 spring-ai 定义一个自动发布mcp-server，打成 jar 包，其通过 stdio 方式与 LLM 进行交互。

Function Call

Function Call 是一种强大的机制，它赋予了大型语言模型（LLM）直接调用外部函数的能力，这就像为模型配备了一把“瑞士军刀”。它的核心思想是让模型能够理解并生成调用特定函数所需的参数，并将函数执行的结果融入自身的响应中。

sequenceDiagram
    participant 用户
    participant 大模型
    participant 外部函数
用户->>大模型: 提问"北京今日气温？"
activate 大模型
大模型->>大模型: 解析意图生成参数
大模型->>大模型: 生成{"location": "北京"}
大模型->>外部函数: 调用get_weather(参数)
activate 外部函数
外部函数-->>大模型: 返回JSON数据
deactivate 外部函数
大模型->>大模型: 数据格式转换
大模型-->>用户: 整合回复"北京今日18摄氏度, 湿度60%"
deactivate 大模型

Function Call 的最大特点在于，它是一种“代码级”的扩展，其紧密集成于模型本身的训练和部署流程中。它不是一个独立的外部服务，而是模型能力的一部分，因此得名“瑞士军刀”，因为它以一种灵活、集成的方式，极大地扩展了模型的应用范围和实用性。

Agent

Agent 是一种具备高度自主性的 AI 实体。它能够独立理解复杂的目标，并将其分解为一系列可执行的子任务。为了完成这些任务，Agent 能够智能地调用各种外部工具，例如利用 MCP Server 访问外部数据源，或通过 Function Call 执行特定代码。这种强大的自主规划和工具调用能力，使得 Agent 能够处理诸如“撰写AI趋势报告”这类需要多步操作和跨工具协作的复杂任务。

以下为基于Agent 完成总结知乎关于 AI 最新讨论任务的例子：

sequenceDiagram
    participant User
    participant LLM
    participant Function
    participant MCP_Server
User->>LLM: 帮我总结知乎上关于AI的最新讨论
activate LLM
LLM->>Function: 调用 `detect_platform(url)`
activate Function
Function-->>LLM: 返回 "知乎"
deactivate Function
LLM->>MCP_Server: 通过 MCP 协议请求爬虫服务
activate MCP_Server
MCP_Server-->>LLM: 返回结构化网页数据
deactivate MCP_Server
LLM-->>User: 生成摘要报告
deactivate LLM
